# 文档生成策略

## 核心规则

**除非用户明确要求生成文档，否则不要生成文档。**

## 原因

- 文档生成会消耗大量 token
- 用户可能只需要核心功能实现，不需要额外的说明文档
- 过多的文档会造成信息噪音

## 何时生成文档

只在以下情况下生成文档：

1. **用户明确要求**
   - "请生成文档"
   - "写一个 README"
   - "创建使用说明"

2. **用户询问是否需要文档**
   - 完成核心功能后，可以询问："需要我生成使用文档吗？"
   - 等待用户确认后再生成

## 何时不生成文档

- 创建脚本或工具时，不要自动生成配套文档
- 修复问题时，不要生成问题总结文档
- 实现功能时，不要生成详细的设计文档（除非用户要求）

## 最佳实践

1. **完成核心工作后简短总结**
   - 用 2-3 句话说明完成了什么
   - 不要创建 markdown 文件来总结

2. **询问而不是假设**
   - 如果觉得文档可能有用，先询问用户
   - 不要假设用户需要文档

3. **保持简洁**
   - 代码注释应该简洁明了
   - 避免过度文档化

## 示例

❌ **错误做法**：
```
我创建了脚本 X，现在让我生成：
- README.md
- USAGE_GUIDE.md  
- TROUBLESHOOTING.md
- EXAMPLES.md
```

✅ **正确做法**：
```
我创建了脚本 X 来解决问题 Y。

使用方法：
.\script.ps1

需要我生成详细的使用文档吗？
```
