# Q4: IM 系统如何保证消息不乱序？

> 2026-02-27

## 结论

不能只靠 Snowflake ID 排序，也不能只靠时间戳。IM 消息排序通常需要一个**会话级别的逻辑序列号（sequence）**，由服务端在同一会话维度内严格递增分配。

---

## 1. 为什么 Snowflake ID 不能直接用来排序？

上一个 Q&A 已经分析过：同一毫秒内不同机器生成的 ID 大小取决于 worker-id 而非生成顺序。

IM 场景下的具体问题：

```
用户A 发消息 → 落到机器1 (worker=1) → ID=100000004096
用户B 发消息 → 落到机器2 (worker=2) → ID=100000008192  (同一毫秒)

实际上 A 先发，但 B 的 ID 更大
```

如果用 ID 排序，消息顺序就错了。

## 2. 时间戳能用来排序吗？

部分能，但不够。

| 问题 | 说明 |
|------|------|
| 精度不够 | 毫秒级时间戳，同一毫秒内多条消息无法区分先后 |
| 时钟不同步 | 不同客户端/服务器的时钟有偏差，A 的 10:00:00.000 可能是 B 的 10:00:00.050 |
| 不可靠 | 客户端时间可以被用户修改 |

时间戳可以作为辅助排序字段（粗粒度），但不能作为唯一排序依据。

## 3. 业界主流方案：会话级逻辑序列号（seq）

核心思路：**在每个会话（conversation）维度内，由服务端分配一个严格递增的整数序列号。**

```
消息表设计：
┌──────────┬─────────────────┬─────┬───────────────┬─────────┐
│ msg_id   │ conversation_id │ seq │ send_time     │ content │
│ (雪花ID) │ (会话ID)        │ (↑) │ (服务端时间戳) │         │
├──────────┼─────────────────┼─────┼───────────────┼─────────┤
│ 10004096 │ conv_001        │  1  │ 1709012345000 │ "你好"  │
│ 10008192 │ conv_001        │  2  │ 1709012345000 │ "在吗"  │
│ 10012288 │ conv_001        │  3  │ 1709012346000 │ "收到"  │
└──────────┴─────────────────┴─────┴───────────────┴─────────┘
```

- `msg_id`（Snowflake）：全局唯一标识，用于去重、索引、跨会话引用
- `seq`：会话内排序的唯一依据，严格递增，无间隔
- `send_time`：展示用，不参与排序逻辑

### seq 怎么生成？

这正好可以用 id-generator 的 Segment 模式：

```java
// 每个会话一个 bizTag，保证会话内严格递增
long seq = idGeneratorClient.nextSegmentId("conv_001");
```

或者更常见的做法是用 Redis INCR（更轻量）：

```
INCR msg:seq:{conversation_id}
```

## 4. 各字段的职责分工

| 字段 | 生成方式 | 职责 | 排序用？ |
|------|---------|------|---------|
| msg_id | Snowflake | 全局唯一标识、去重、跨会话引用 | 否 |
| seq | Segment / Redis INCR | 会话内排序 | 是（唯一依据） |
| send_time | 服务端时间戳 | UI 展示时间、粗粒度查询 | 辅助 |

## 5. 客户端怎么用 seq 保证顺序？

客户端本地记录当前会话已收到的最大 seq，拉取/推送时按 seq 排序插入：

```
客户端本地 max_seq = 5

收到推送 seq=7 → 发现缺 seq=6 → 向服务端补拉 seq=6
收到推送 seq=6 → 插入，连续了 → 按 seq 排序展示
```

这也是微信、钉钉等 IM 的通用做法——seq 既是排序依据，也是增量同步的游标。

## 6. 总结

回到你的问题：

- 不是用另一个时间戳字段来排序——时间戳精度不够、时钟不可信
- 正确做法是引入一个**会话级逻辑序列号（seq）**，由服务端在每个会话维度内严格递增分配
- Snowflake ID 负责「全局唯一」，seq 负责「会话内有序」，各司其职
- seq 的生成可以用 id-generator 的 Segment 模式（按 conversation_id 做 bizTag），也可以用 Redis INCR（更轻量）
